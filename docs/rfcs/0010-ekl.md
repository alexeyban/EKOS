# RFC 0010 — Enterprise Knowledge Language (EKL)

**Status:** Accepted
**Date:** 2026-07-11

(Numbered 0010, not 0009 — 0009 was taken by the AI Runtime RFC before this one was written.)

---

## Problem

`ekos ask` (RFC 0009) answers natural-language questions but its retrieval path is a fuzzy FTS5
match — good for "what does X depend on" style questions, bad for precise, composable, scriptable
queries. Power users and integrations (dashboards, CI checks, other tools) need a deterministic
query language: given the same ledger state and the same query text, always the same result set,
with no LLM in the loop. EKL is that language — the SQL-equivalent for the EKOS knowledge graph.

---

## Scope for v0

EKL v0 supports flat predicate queries over the two enumerable entities in the ledger (`Object`,
`Relationship`), optionally anchored to a named starting object's immediate neighbourhood. It does
**not** support arbitrary multi-hop path expressions in v0 — see Non-goals. This is a deliberate
narrowing from the original phase framing so that what ships is fully specified and testable rather
than partially implemented.

### Grammar (EBNF)

```
query        := "FIND" entity whereClause? fromClause? returnClause? orderClause? limitClause? ;
entity       := "Object" | "Relationship" ;
whereClause  := "WHERE" predicate ( "AND" predicate )* ;
predicate    := field op literal ;
op           := "=" | "!=" | ">" | "<" | ">=" | "<=" | "CONTAINS" ;
fromClause   := "FROM" string ;
returnClause := "RETURN" field ( "," field )* ;
orderClause  := "ORDER" "BY" field ( "ASC" | "DESC" )? ;
limitClause  := "LIMIT" number ;
field        := identifier ;
literal      := string | number ;
string       := "'" <any char except "'"> * "'" ;
number       := digit+ ( "." digit+ )? ;
identifier   := letter ( letter | digit | "_" )* ;
```

Clauses may appear in any order after `WHERE`/`FROM` (parser accepts `RETURN`, `ORDER BY`, `LIMIT`
in any relative order) but the AST is built to the fixed `EklAst` shape below regardless of
surface order.

### Fields

| Entity | Available fields |
|---|---|
| `Object` | `id`, `name`, `kind`, `evidence` (array of evidence ids), `created_at` |
| `Relationship` | `id`, `kind`, `from`, `to`, `created_at` |

`kind` for `Object` compares against `ObjectKind`'s `Display` string (e.g. `Table`). `kind` for
`Relationship` compares against `RelationshipKind`'s `Debug` string (e.g. `ForeignKey`) — the same
representation the ledger already stores in `current_relationships.kind` (see `ledger::append_relationship`).

### Semantics — mapping to Runtime calls

1. **Entity enumeration.** `Object` queries without `FROM` call `Runtime::list_objects()`.
   `Relationship` queries without `FROM` call `Runtime::list_relationships()`. Both are new
   Runtime methods added by this RFC (the ledger has no existing "all rows" query — `find_objects`
   is FTS-ranked search, not enumeration).
2. **`FROM '<name>'` anchor.** The anchor string is resolved via `Runtime::find_objects(name)`,
   taking the top-ranked match (`EklError::AnchorNotFound` if empty). The candidate set becomes
   `Runtime::load_neighborhood(anchor_id, 1).objects` (for `Object` queries) or `.relationships`
   (for `Relationship` queries) — i.e. the anchor plus its direct neighbours/relationships. Depth
   is fixed at 1 in v0.
3. **Row projection.** Every candidate object/relationship is converted to a
   `HashMap<String, serde_json::Value>` row using the field table above, *before* filtering — this
   keeps predicate evaluation generic (`serde_json::Value` equality/ordering) instead of one match
   arm per entity type.
4. **`WHERE` filtering.** Predicates are ANDed. `=`/`!=` compare the field's string/number
   representation against the literal. `>`/`<`/`>=`/`<=` require both sides to parse as `f64`;
   if either side doesn't parse as a number, the predicate evaluates to `false` (not an error) —
   a deliberate v0 simplification so a stray numeric comparison on a string field silently returns
   no rows rather than aborting the whole query. `CONTAINS` does a substring check against the
   field's string representation.
5. **`RETURN`.** Projects the row down to the named fields. If omitted: `Object` defaults to
   `id, name, kind`; `Relationship` defaults to `id, kind, from, to`.
6. **`ORDER BY` / `LIMIT`.** Applied last, over the filtered+projected rows. Missing field or
   non-comparable values sort last (stable).

### Types

```rust
pub struct EklAst {
    pub entity: Entity,
    pub predicates: Vec<Predicate>,
    pub from: Option<String>,
    pub returns: Vec<String>,
    pub order_by: Option<(String, Order)>,
    pub limit: Option<u64>,
}

pub enum Entity { Object, Relationship }
pub struct Predicate { pub field: String, pub op: Op, pub value: Literal }
pub enum Op { Eq, Ne, Gt, Lt, Ge, Le, Contains }
pub enum Literal { Str(String), Num(f64) }
pub enum Order { Asc, Desc }

pub fn ekl_parse(input: &str) -> Result<EklAst, ParseError>;

pub struct EklInterpreter<'a> { /* holds &'a Runtime<'a> */ }
impl<'a> EklInterpreter<'a> {
    pub fn execute(&self, ast: &EklAst) -> Result<EklResult, EklError>;
}
pub struct EklResult { pub rows: Vec<HashMap<String, serde_json::Value>> }
```

Parser is a hand-written lexer + recursive-descent parser (no `pest`/`nom` dependency) — the
grammar above is small enough that a dependency doesn't pay for itself, and it keeps the crate
count and build time down. `ParseError { message: String, position: usize }` reports a byte offset
into the input; the CLI renders it as `^` under the offending token.

```rust

impl Ledger {

    pub fn all_relationships(&self) -> Result<Vec<KirRelationship>, LedgerError>;
}

// ekos-runtime
impl<'a> Runtime<'a> {
    pub fn list_objects(&self) -> Result<Vec<KirObject>, RuntimeError>;
    pub fn list_relationships(&self) -> Result<Vec<KirRelationship>, RuntimeError>;
}
```

Both are thin wrappers, following the exact pattern `find_objects` already established in Phase 10.

## Example queries (ecommerce fixture)

Fixture: `orders` (Table), `customers` (Table), `order_items` (Table), with relationships
`orders --ForeignKey--> customers` and `orders --ForeignKey--> order_items`.

| # | Query | Expected result |
|---|---|---|
| 1 | `FIND Object WHERE kind = 'Table'` | 3 rows: `orders`, `customers`, `order_items` (default columns `id, name, kind`) |
| 2 | `FIND Object WHERE kind = 'Table' RETURN name` | 3 rows, one column `name` |
| 3 | `FIND Object WHERE name = 'orders'` | 1 row: `orders` |
| 4 | `FIND Object WHERE name CONTAINS 'order'` | 2 rows: `orders`, `order_items` |
| 5 | `FIND Object WHERE kind = 'Table' ORDER BY name LIMIT 1` | 1 row: `customers` (alphabetically first) |
| 6 | `FIND Relationship WHERE kind = 'ForeignKey'` | 2 rows (both FKs) |
| 7 | `FIND Relationship WHERE kind = 'ForeignKey' FROM 'orders'` | 2 rows: both FKs from `orders` |
| 8 | `FIND Object FROM 'orders'` | 3 rows: `orders` + its 2 direct neighbours (`customers`, `order_items`) |
| 9 | `FIND Object WHERE kind = 'Service'` | 0 rows (none in fixture) |
| 10 | `FIND Relationship WHERE kind = 'ForeignKey' RETURN from, to LIMIT 1` | 1 row, columns `from`, `to` |

## Non-goals (v0)

- **Path expressions** (`orders -> customer_id -> customers`, multi-hop chained traversal). This
  was in the original phase framing but is deferred — `FROM` + depth-1 neighbourhood covers the
  validation example (`FIND Relationship WHERE kind = 'ForeignKey' FROM 'orders'`) without the
  added grammar/interpreter complexity of arbitrary path chains. A follow-up RFC can extend
  `fromClause` into a full path-expression grammar once there's a concrete multi-hop use case.
- **`DEPTH` clause** — anchor expansion is fixed at 1 hop. Adding a configurable depth is a small
  extension once needed; not adding speculative surface area now.
- **Joins across `Object` and `Relationship` in one query** — each query targets exactly one
  entity type.
- **Aggregation** (`COUNT`, `GROUP BY`) — not needed yet; `ekos ekl "..." | wc -l` covers counting
  for now.
- **Historical queries** (`AS OF <timestamp>`) — EKL v0 only queries current state, matching
  `Runtime::load_object`/`load_neighborhood`, not `object_at`/`relationships_at`.

---

## Alternatives considered

**Reuse `pest`/`nom` for the parser** — rejected for v0. The grammar is small (six clause types,
no recursive expression precedence beyond flat `AND`-chains), and a hand-written recursive-descent
parser is easy to fuzz-test for "never panics, only returns `ParseError`" — which is the actual
acceptance bar, not grammar elegance. Revisit if the grammar grows (e.g. path expressions, `OR`,
parenthesized groups) to the point where a hand-written parser becomes unwieldy.

**Compile EKL to SQL and run directly against the ledger's SQLite tables** — rejected. This would
bypass `Runtime` entirely, breaking the invariant that all knowledge consumers go through the
Runtime (RFC 0005), and would leak the ledger's internal schema into query semantics.

---

## Acceptance Criteria

- [ ] `ekl_parse` produces a correct `EklAst` for all 10 example queries above.
- [ ] `ekl_parse` never panics on arbitrary input — malformed input returns `ParseError`.
- [ ] `EklInterpreter::execute` against the ecommerce fixture produces the expected row counts and
      values for all 10 example queries.
- [ ] `ekos ekl "<query>"` prints results as a table by default, `--json` prints `EklResult` as JSON.
- [ ] A `ParseError` from the CLI prints the offending position, not a Rust panic/backtrace.
