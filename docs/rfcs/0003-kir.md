# RFC 0003 — Knowledge Intermediate Representation (KIR)

**Status:** Accepted  
**Author:** alexeyban  
**Created:** 2026-07-02  
**Gating:** Phase 5

---

## Motivation

The Knowledge Compiler needs a stable, serializable, graph-structured intermediate representation
that sits between raw observations (connector output) and the Canonical Knowledge Model (Phase 8+).
KIR must be:

- **Pure structural** — no semantic enrichment, no merging, no inference. Those are later passes.
- **Evidence-anchored** — every node traces to at least one `KirEvidence` record.
- **Serializable** — stored as `KnowledgeArtifact` JSON blobs in the artifact store.
- **Connector-neutral** — both a SQL schema analysis and a git history produce KIR; they are not
  distinguished at the KIR layer.

---

## Design

### Four primitives

| Type | Role | Key fields |
|---|---|---|
| `KirObject` | Identity node (entity, table, file, service …) | `id`, `name`, `kind`, `properties`, `evidence[]` |
| `KirRelationship` | Directed semantic edge between two objects | `id`, `kind`, `from`, `to`, `properties`, `evidence[]` |
| `KirEvent` | Immutable change record (commit, deploy, migration …) | `id`, `kind`, `subject`, `payload`, `evidence[]`, `occurred_at` |
| `KirEvidence` | Provenance pointer (source file + line + fragment) | `id`, `location`, `fragment`, `confidence` |

### `KirId`

`KirId` wraps a `Uuid`. Passes that produce deterministic output **must** use UUIDv5 derived from
stable content (path, SQL DDL hash, commit SHA). Passes using external non-deterministic sources
(LLM, wall-clock) may use UUIDv4 temporarily; the Identity Resolution pass (Phase 7) will merge
synonymous nodes.

### `KirGraph`

A bag-of-nodes container:

```rust
pub struct KirGraph {
    pub objects: Vec<KirObject>,
    pub relationships: Vec<KirRelationship>,
    pub events: Vec<KirEvent>,
    pub evidence: Vec<KirEvidence>,
}
```

`KirGraph` is the payload of a `KnowledgeArtifact`. One compiler pass emits one `KnowledgeArtifact`
whose content is the pass's full `KirGraph` output plus the input artifact IDs.

### No semantic enrichment in this layer

The `kir` crate must **not** import or depend on identity resolution, semantic, or LLM crates.
It is a pure data structure crate. All enrichment happens in later compiler passes.

---

## Alternatives Considered

- **Single `KirNode` enum instead of four types** — rejected; homogeneous representation loses
  structural type information and makes pattern matching on the consuming side verbose.
- **RDF triples** — rejected; triples cannot carry structured payload on edges or events.

---

## Acceptance Criteria

- [ ] All four types compile with `Serialize + Deserialize`.
- [ ] Round-trip: `serde_json::to_string(&graph)` → `serde_json::from_str(…)` produces equal graph.
- [ ] `KirGraph` can be embedded in `KnowledgeArtifact::content.kir`.
- [ ] `cargo clippy -p ekos-kir` produces zero errors.
- [ ] `kir` crate imports zero crates from `compiler-core`, `identity`, `semantic`, or `recovery`.
